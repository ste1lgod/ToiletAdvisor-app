## Словарь терминов

| Термин | Что это такое |
|--------|--------------|
| **Firestore** | База данных в облаке (Firebase). Хранит туалеты, отзывы, пользователей, логи, избранное. Данные в реальном времени синхронизируются между устройствами. |
| **localStorage** | Локальное хранилище в браузере пользователя. Данные сохраняются между перезагрузками страницы, но только на одном устройстве. Используется как быстрый кэш. |
| **Кэш** | Временное хранилище данных для ускорения. Вместо того чтобы каждый раз ходить в базу — сначала смотрим в кэш. Если данные там свежие — возвращаем сразу. |
| **TTL (Time To Live)** | "Срок жизни" кэша. Например TTL = 30 минут означает что через 30 минут кэш считается устаревшим и нужно перезагрузить из базы. |
| **Batch** | Пакетная операция в Firestore. Вместо 50 отдельных запросов к базе — собираем все изменения в один пакет и отправляем за раз. Быстрее и надёжнее. |
| **userId** | Уникальный идентификатор пользователя в базе. Например `admin`, `admin2`, `JuOdNQRuMUvY6P4AAU5Z`. |
| **userPhone** | Поле в отзыве которое хранит отображаемое имя автора. Несмотря на название — там может быть ник, телефон или логин. |
| **Геокодирование** | Превращение координат (41.308, 69.259) в читаемый адрес ("Афрасиаб улица"). Делается через сервис Nominatim (OpenStreetMap). |
| **Rate limit** | Ограничение на частоту запросов. Nominatim разрешает не более 1 запроса в секунду — иначе блокирует. |
| **Оптимистичный рендер** | Показываем результат сразу в UI не дожидаясь ответа сервера. Например отзыв появляется в списке мгновенно, а реально сохраняется в базе чуть позже. |
| **Stale-while-revalidate** | Стратегия кэширования: показываем старые данные мгновенно, а в фоне тихо обновляем из базы. Когда новые данные придут — обновляем UI без задержки. |
| **Inval idация кэша** | Принудительное удаление кэша. После изменения данных (например смена ника) — старый кэш больше не актуален, его нужно удалить чтобы при следующем запросе загрузились свежие данные. |
| **Race condition** | Ситуация когда два процесса работают параллельно и мешают друг другу. Например два запроса геокодирования одной точки одновременно. |
| **DOM** | Document Object Model — структура HTML-страницы в браузере. Когда мы "обновляем DOM" — мы меняем что видит пользователь на экране. |
| **Seed-данные** | Начальные тестовые данные которые были загружены в базу при первом запуске. Туалеты `seed-0001` … `seed-0007` — это они. |
| **Дубликаты** | Несколько одинаковых записей в базе. В нашем случае — 12 аккаунтов с одним телефоном `+998888`, созданных seed-скриптом случайно. |

## ЧАСТЬ 2: Что исправлено в коде

### 2.1 Критический баг в статистике — `ReferenceError` 🔴→✅

**Файл:** `js/admin.js`

**Проблема:** В функции `loadAdminStats` (загружает страницу статистики в панели admin) переменная `reviewsByUser` (словарь "сколько отзывов у каждого пользователя") объявлялась ПОСЛЕ того как использовалась в фильтре дубликатов. В JavaScript нельзя использовать переменную до её объявления — это называется `ReferenceError` и приводило к тому что страница статистики пользователей падала с ошибкой.

**Было:**
```js
// Фильтруем дубликатов...
const dedupUsers = users.filter(u => {
  if(reviewsByUser[u.id] > 0) return true;  // ← ОШИБКА: reviewsByUser ещё не объявлен!
  ...
});
...
const reviewsByUser = {};  // ← объявлен ПОСЛЕ использования
```

**Стало:**
```js
const reviewsByUser = {};  // ← сначала объявляем
reviews.forEach(r => { reviewsByUser[r.userId] = (reviewsByUser[r.userId]||0)+1; });

// Теперь можно использовать
const dedupUsers = users.filter(u => {
  if(reviewsByUser[u.id] > 0) return true;  // ← теперь работает
});
```

**Дополнительно:** В счётчиках статистики теперь показывается `dedupUsers.length` (7) вместо `users.length` (19 с дубликатами).

### 2.2 Двойное геокодирование — лишние запросы к Nominatim 🟡→✅

**Файлы:** `js/init.js`, `js/map.js`

**Проблема:** Были две функции которые обе геокодировали (переводили координаты в адреса) одни и те же точки без адреса. Они запускались одновременно при старте приложения.

- `_patchSeedAddresses()` в `init.js` — шаг 1: записывал seed-адреса, шаг 2: геокодировал остальные
- `_geocodeMissingAddresses()` в `map.js` — тоже геокодировал точки без адреса

Из-за этого Nominatim получал 2 запроса для одной точки почти одновременно. Nominatim разрешает только 1 запрос в секунду — лишние запросы блокировались с ошибкой 429.

**Исправлено:** Из `_patchSeedAddresses` убран шаг 2 (геокодирование). Теперь `_patchSeedAddresses` только записывает уже известные адреса seed-точек в Firestore. Всё геокодирование делает только `_geocodeMissingAddresses`, с задержкой 1100ms между запросами.

### 2.3 Баг с адресом при добавлении в избранное 🟡→✅

**Файл:** `js/favorites.js`

**Проблема:** При добавлении туалета в избранное адрес брался из поля `toilet.addr` без проверки — годный ли он. Если там лежали координаты в виде текста (`"41.3082, 69.2598"`) или текст совпадал с описанием туалета — этот плохой адрес записывался в избранное.

**Добавлена функция `_isBadAddr(addr, desc)`** — проверяет адрес на "плохость":
- пустой адрес → плохой
- адрес = `"Нет адреса"` или `"—"` → плохой  
- адрес совпадает с описанием туалета → плохой
- адрес выглядит как координаты (`"41.308, 69.259"`) → плохой

Теперь если адрес плохой — запускается геокодирование через Nominatim и в избранное записывается нормальный адрес.

### 2.4 Баг с DOM-индексом в избранном — обновлялся не тот элемент 🟡→✅

**Файл:** `js/favorites.js`

**Проблема:** Функция `_fixFavAddresses` (исправляет адреса в уже отображённом списке избранного) обновляла элемент на экране по его **порядковому номеру** в массиве. Если пока шло геокодирование (это занимает время) пользователь добавил или удалил элемент из избранного — порядковые номера сдвигались и обновлялся не тот элемент.

**Исправлено:** Каждый элемент списка избранного теперь получает атрибут `data-fav-id="ID_туалета"`. Поиск элемента идёт по этому атрибуту — надёжно вне зависимости от порядка.

**Было:** `items[idx].querySelector('.pFavItem-addr')`  
**Стало:** `document.querySelector('#pFavList .pFavItem[data-fav-id="${f.id}"] .pFavItem-addr')`

### 2.6 Ошибка при сохранении отзыва — неверный toast 🟡→✅

При этом оптимистично добавленный элемент в списке отзывов оставался — пользователь видел свой отзыв, но при обновлении страницы он пропадал.

### 2.7 Кэш пользователей — несколько проблем 🟡→✅

**Файл:** `js/profile.js`

**Что такое кэш пользователей:** При открытии отзыва нужно показать имя автора. Но делать запрос к базе для каждого отзыва — медленно. Поэтому один раз загружаем всех пользователей и храним в памяти (`_usersCache`) и в `localStorage`. При следующей загрузке страницы — восстанавливаем из `localStorage` мгновенно, без ожидания.

**Проблема 1:** `_restoreUsersCache` (восстановление кэша из localStorage при старте) не проверял тип данных. Если по какой-то причине в localStorage оказался битый объект (например массив вместо словаря) — он мог подгрузить его и сломать поиск имён.

**Исправлено:** Добавлена проверка `typeof stored.data === 'object' && !Array.isArray(stored.data)`.

**Проблема 2:** `_getActualName` (функция которая возвращает имя пользователя для отображения) возвращала `'?'` если имя не найдено. Это выглядело странно в UI.

**Исправлено:** Возвращает `'Пользователь'` вместо `'?'`.

**Проблема 3:** После регистрации нового пользователя — он не попадал в `_usersCache` сразу. Если пользователь сразу оставлял отзыв — в кэше его не было и показывался телефон вместо ника.

**Исправлено:** После регистрации сразу добавляем пользователя в кэш.

**Проблема 4:** После входа администратора — кэш не обновлялся с актуальными данными admin (ник мог устареть).

**Исправлено:** После входа обновляем запись admin в кэше.

### 2.8 `saveNick` — несколько проблем 🟡→✅

**Файл:** `js/profile.js`

**Что делает `saveNick`:** Когда пользователь вводит ник в профиле и уходит с поля (нажимает на другое место) — функция сохраняет ник в базу и обновляет его во всех отзывах и логах этого пользователя.

**Проблема 1:** Функция запускалась при каждом уходе с поля (`onblur`) — даже если пользователь кликнул на поле и сразу вышел ничего не меняя. Каждый раз делалось 3 запроса к Firestore.

**Исправлено:** В начале функции проверяем — изменился ли ник. Если нет — выходим сразу.

**Проблема 2:** `batch.commit()` вызывался даже если отзывов и логов у пользователя нет — лишний сетевой запрос.

**Исправлено:** Batch создаётся только если есть что обновлять.

**Проблема 3:** Инвалидация кэша (удаление устаревшего кэша) вызывалась только когда ник непустой. Если пользователь **удалял** ник — кэш не обновлялся и везде показывалось старое имя.

**Исправлено:** Инвалидация кэша происходит всегда при любом изменении ника.

**Проблема 4:** После сохранения ника не было никакого уведомления пользователю.

**Исправлено:** Показывается toast "Ник сохранён ✓" или "Ник удалён".

### 2.9 `_patchReviewNicks` — retry при ошибке 🟡→✅

**Файл:** `js/init.js`

**Что делает:** При старте приложения проходит по всем отзывам и обновляет в них имя автора на актуальный ник. Запускается один раз в 5 секунд после загрузки.

**Проблема:** При ошибке сети флаг `_patchReviewNicksDone` сбрасывался, но повторного запуска не происходило — в этой сессии патч больше не запускался.

**Исправлено:** При ошибке — автоматический повторный запуск через 30 секунд.

### 2.10 Stale-while-revalidate — улучшено сравнение 🟡→✅

**Файл:** `js/sheet.js`

**Что такое:** Стратегия загрузки отзывов. При открытии шторки туалета — сначала мгновенно показываем отзывы из кэша. Параллельно тихо загружаем свежие из базы. Если что-то изменилось — обновляем список без мигания.

**Проблема:** Для определения "изменилось ли что-то" сравнивались только `id` и `userPhone` каждого отзыва. Если пользователь изменил **текст** отзыва или **рейтинг** — это изменение не замечалось и список не обновлялся.

**Исправлено:** Теперь сравниваются `id + userPhone + rating + первые 20 символов текста`.

### 2.11 `_saveReviewsToStorage` — обработка переполнения 🟡→✅

**Файл:** `js/sheet.js`

**Проблема:** `localStorage` имеет ограничение ~5MB. Если оно исчерпано — `localStorage.setItem` выбрасывает ошибку `QuotaExceededError`. Эта ошибка молча игнорировалась и кэш отзывов не сохранялся — при следующем открытии всегда шёл запрос к базе.

**Исправлено:** При переполнении — очищаем все старые кэши отзывов (`ta_rv_*`) и пробуем ещё раз.

### 2.12 Кнопки Назад/Обновить — выравнивание 🟡→✅

**Файлы:** `css/admin.css`, `css/ui.css`

Также убрали SVG-иконку стрелки из кнопки "Обновить" — осталось только слово. Исправили асимметричный `padding` кнопки который был под иконку.
